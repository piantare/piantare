## Achados

Severidade: **blocker** > **high** > **medium** > **low**.

### #1 — `<Link>` "Criar conta" não troca o modo do form

- **Tipo:** BUG
- **Severidade:** high
- **Onde:** `/login` (página de auth), na transição "Entrar" ↔ "Criar conta"
- **Sintoma:** Cliquei no link "Criar conta" cuja `href="/login?mode=signup"`. A URL ficou em `/login` sem o query string e a tela continuou em modo "Entrar". Funciona corretamente quando navego diretamente para `/login?mode=signup`.
- **Hipótese:** O componente `/login/page.tsx` é server component que lê `searchParams`. O Next.js Link client-side roteia para a mesma rota (`/login`) e o React não re-fetcha o RSC porque o pathname não mudou — só o query.
- **Fix sugerido:** Trocar `<Link>` por `<a>`, ou marcar a página como dinâmica e usar `useSearchParams` no client, ou usar `router.push` com `router.refresh()`.

### #2 — Email confirmation ainda estava ON no Supabase ✅ RESOLVIDO

- **Tipo:** BLOCKER (resolvido em runtime)
- **Severidade:** blocker
- **Onde:** Supabase Auth config do projeto `ydmcyituvsjnfagxjgka`
- **Sintoma:** Signup retornou `needsConfirmation=true` e login subsequente deu "Email not confirmed".
- **Resolução:** Nathan flipou o toggle no Supabase dashboard durante o smoke test. Users pendentes foram auto-confirmados.
- **Follow-up:** Documentar em `docs/DEPLOY.md` como pré-requisito de go-live para qualquer ambiente futuro.

### #3 — Form submit via botão click às vezes não dispara

- **Tipo:** FRICTION (provavelmente MCP-only, mas vale validar)
- **Severidade:** medium (alto se for bug real, low se MCP-only)
- **Onde:** Forms de `/login` e `/login?mode=signup`
- **Sintoma:** Click no botão "Entrar" / "Criar conta" às vezes não submete o form. Pressionar `Enter` no campo de senha sempre funcionou.
- **Hipótese A (MCP):** O click no botão pode estar acionando o handler do elemento sem disparar o `submit` semântico do form, ou o read_page intermediário pode estar invalidando o action token do form server action.
- **Hipótese B (real):** Algum hidden field stale (e.g. CSRF/redirectTo token) pode falhar a validação server-side silenciosamente em alguns cenários.
- **Próximo passo:** Validar manualmente via teclado vs mouse em browser real.

### #4 — Moeda decorativa / inconsistente

- **Tipo:** FRICTION
- **Severidade:** medium
- **Onde:** Onboarding (`/onboarding`), criação de produto (`/lab/products/new`), criação de pedido
- **Sintoma:**
  - Default Moeda=USD com País=BR é incoerente.
  - O campo Moeda da organização parece **decorativo**: produtos têm `price_usd` e pedidos têm `unit_price_usd`/`total_usd` hardcoded em USD. A Apoteca Vitalis foi criada com BRL e os pedidos dela continuam em US$.
- **Fix sugerido (MVP):** Esconder o seletor de moeda do form de onboarding até multi-currency real entrar no escopo. Hardcode USD por enquanto e deixar claro no rótulo do preço ("Preço (USD)" já está OK).

### #5 — RLS bloqueia leitura cross-side de `organizations` (impacto: marketplace quebrado)

- **Tipo:** BUG
- **Severidade:** **blocker para parceiros reais**
- **Onde:** `/brand/catalog`, `/orders`, `/orders/[id]`
- **Sintoma observado:**
  - No catálogo do brand, "Lab:" aparece vazio (não mostra "Cannalab Sul")
  - Na inbox do brand, "Lab:" vazio
  - Na inbox do lab, "Brand:" vazio
  - Na página de detalhe do pedido: lado próprio preenchido, lado contraparte vazio
- **Root cause:** A policy SELECT em `organizations` provavelmente só permite leitura de orgs onde o user é membro. Quando o pedido faz embedding via PostgREST FK (`brand:organizations!brand_id(name)`), o lado da contraparte é filtrado pelo RLS e retorna `null`.
- **Fix sugerido (opções):**
  - **(a)** Denormalizar: gravar `brand_name_snapshot` e `lab_name_snapshot` em `orders` no `create-order` e no catalogo. Sem RLS hassle, leve drift se org renomear.
  - **(b)** Adicionar policy: "permite ler o nome de qualquer org que esteja em pedido ou produto onde você é contraparte". Mais correto, mas RLS recursivo costuma ser frágil.
  - **(c)** Mover as leituras de catalogo + pedidos para o admin client (service-role), e fazer o gating de autorização no app layer. Simples, mas precisa cuidado para não vazar.
- **Recomendação:** (a) para o MVP, (b) quando houver fôlego para revisitar RLS.

## Leitura subjetiva — "o que o usuário sente em cada etapa"

### Landing (`/` → `/login`)
**Frio.** Caixa branca isolada no meio da tela, palavra "Piantare" sem logo, tagline, contexto. Para parceiro real que chega via WhatsApp/email não tem problema (já sabe o que é). Para qualquer outro, é seco. Não comunica longevidade, não comunica saúde, não comunica nada. **Não é blocker pra parceiros fechados, é blocker se a URL vazar.**

### Signup / Login
**Funcional, sem alma.** Email + senha + botão. Faltam pequenos sinais que reduziriam ansiedade: indicador de "criando conta...", confirmação visual além de redirect, link "Esqueci minha senha" (não existe — bloqueia 100% dos users que esquecerem). O bug do link "Criar conta" cria momento de "ué, cliquei no lugar errado?".

### Onboarding (criar org)
**Boa intenção, execução enxuta.** A copy "Você precisa de um lab (quem produz) ou um brand (quem encomenda) para usar a plataforma. Pode trocar depois." é a melhor frase de toda a UI até aqui — explica o sistema em uma linha. Faltam: exemplos visuais do que é cada um, ou pelo menos um link "Como funciona?". Campo Moeda atrapalha mais do que ajuda.

### Lab — primeira tela `/lab/products`
**Empty state bem feito.** "Nenhum produto ainda. Crie o primeiro para aparecer no catálogo dos brands." é claro, conecta produto a próximo passo. CTA "Novo produto" bem destacado.

### Criar produto
**Fricção quase zero.** 4 campos, todos óbvios. Placeholder "ex.: kg, frasco, dose" é boa orientação. Falta: validação de preço (aceita 0 ou negativo? não testei). Falta: preview antes de salvar.

### Catálogo do brand
**Aqui dói.** Ver "Lab:" vazio no card do único produto destrói a confiança imediatamente. Parceiro real pensaria "sistema bugado". É o achado de maior impacto subjetivo.

### Criar pedido
**Rápido mas frio.** Dropdown + quantidade + condições + submit. Não há resumo antes de enviar (quanto vai dar de total?), não há confirmação de "este pedido vai para Cannalab Sul, ok?". Cálculo de total só aparece DEPOIS de criar o pedido.

### Página do pedido
**A melhor tela do MVP.** Status machine visual (chips coloridos) + texto educacional ("Apenas o lab pode avançar o status.", "A invoice é gerada automaticamente...") + ações contextuais. Aqui o produto fica vivo. Faltam: timestamps de cada transição ("Aprovado em 21/05 13:42"), histórico/audit log, mensagens entre as partes.

### Transição de status
**Satisfatório.** Click → next status com novo botão aparecendo. Loop fechado bonito quando chega em "Enviado" + "Pedido finalizado.". A invoice aparecendo automaticamente após "Aprovado" é deleitoso.

### Marcar invoice paga
**Honesto.** Botão diz "(mock)" — não tenta enganar. Click → status "pago" sem cerimônia. Para o MVP, suficiente.

### Sensação geral
- **Tempo até primeiro valor percebido:** ~90 segundos (signup → criar org → ver "Produtos" vazia).
- **Tempo até loop completo:** ~3 minutos para um operador rápido.
- **Confiança:** baixa nos primeiros 60 segundos (UI muda, lab name vazio, sem signals de "estou no lugar certo"). Cresce ao avançar para a tela do pedido, que é onde o produto começa a se mostrar.
- **Recorrência:** zero motivos hoje para um lab voltar amanhã. Faltam dashboards, alertas, métricas, ou pelo menos um "1 novo pedido" na home.
- **Emoção:** neutra. O produto cumpre o loop, não emociona. Para alinhar com a visão "longevidade · relacional · longitudinal", precisa de pelo menos um momento humano — talvez na confirmação de pedido criado ("Sua primeira ponte está aberta com [Lab]"), ou ao receber o primeiro pedido como lab.
